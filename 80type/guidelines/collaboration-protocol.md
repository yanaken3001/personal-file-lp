# 連携プロトコル＆エスカレーションルール

## 部門間連携ルール

### デザイン → 開発
- hana/renがデザインを作成 → yuto/meiが実装可能性を確認
- 実装上の制約がある場合、代替案を提示して協議
- デザインの変更は必ずhanaを通す（直接CSSで調整しない）

### コンテンツ → デザイン
- mio/tomoがテキストを作成 → hanaがレイアウトに組み込み
- テキスト量がレイアウトに収まらない場合、mio/tomoに短縮版を依頼

### マーケ → 開発
- shin/aoiが施策を企画 → yuto/kaiが技術的実現性を確認
- A/Bテスト実装はyutoが担当

### 全部門 → 品質管理
- アウトプット完成後、junがレビュー
- Critical/Major指摘は修正必須、Minor/Suggestionは任意

## エスカレーションルール

### 自分で判断してよいこと
- 自分の専門領域内の技術的判断
- ガイドライン通りの実行
- 既存仕様の範囲内での改善

### 他エージェントに相談すべきこと
- 他部門の専門領域に影響する変更
- 既存仕様の解釈が曖昧な場合
- 複数の実装方法があり、どちらがベストか判断できない場合

### 代表に確認必須のこと
- 確定済み仕様の変更（タイプ名、カラー、相性スコア等）
- 新機能の追加・削除
- 予算に影響する判断（有料サービス導入等）
- ブランドに影響する判断
- セキュリティ・個人情報に関する判断
- 外部サービスとの連携開始

## 成果物の保存ルール
- 各エージェントの成果物はファイルとして保存する
- ファイル名：`[部門番号]-[担当名]-[内容]-[日付].md`
- 例：`2-hana-result-screen-design-20260328.md`
- 大型タスクではフェーズごとに成果物を保存し、次フェーズのエージェントはそのファイルを読んで作業する
